# Техническое задание: Система управления подписками гольф-клуба

## 1. Общее описание проекта

### 1.1 Название проекта
Система мониторинга и управления подписками гостей гольф-клуба с интеграцией Telegram-бота

### 1.2 Цель проекта
Создание автоматизированной системы для управления подписками гостей гольф-клуба, включающей администраторскую панель и клиентский Telegram-бот для удобного бронирования услуг и управления подписками.

### 1.3 Текущее состояние
- Проект создан на фреймворке Laravel
- Установлен и настроен пакет Laravel Orchid для администраторской панели
- Бизнес-логика не реализована

---

## 2. Бизнес-требования

### 2.1 Описание бизнес-процесса

Гольф-клуб предоставляет два вида услуг для своих гостей:

1. **Подписка на игру в гольф**
   - Единоразовая подписка (одноразовый доступ)
   - Ежемесячная подписка (регулярный платеж)

2. **Бронирование шкафа для оборудования**
   - Минимальный срок бронирования: 1 месяц
   - Фиксированная стоимость: $10 в месяц
   - Количество шкафов ограничено

### 2.2 Типы пользователей

1. **Администраторы**
   - Управление базой клиентов
   - Подтверждение регистраций новых клиентов
   - Обработка запросов на бронирование
   - Проверка платежей
   - Управление доступными шкафами
   - Мониторинг активных подписок

2. **Клиенты (гости гольф-клуба)**
   - Регистрация через Telegram-бот
   - Просмотр своих активных подписок
   - Бронирование услуг
   - Отправка чеков об оплате
   - Получение уведомлений

## 3. Функциональные требования

### 3.1 Администраторская панель (Laravel Orchid)

#### 3.1.1 Управление клиентами
- База данных существующих клиентов
- Формат номера телефона: **+998 xx xxx-xx-xx** (строгий шаблон)
- Карточка клиента должна содержать:
  - ФИО клиента
  - Номер телефона (уникальный идентификатор)
  - Дата регистрации
  - Статус (новый/подтвержденный/заблокированный)
  - История подписок
  - История платежей
  - Активные подписки

#### 3.1.2 Обработка запросов на регистрацию
- Очередь ожидающих подтверждения клиентов
- Информация о новом клиенте (номер телефона, имя из Telegram)
- Действия: принять/отклонить
- При принятии клиент автоматически добавляется в базу подтвержденных

#### 3.1.3 Обработка запросов на бронирование
- Просмотр всех входящих запросов
- Информация о запросе:
  - Клиент (ФИО, номер телефона)
  - Тип услуги (игра/шкаф/оба)
  - Срок подписки
  - Стоимость
  - Дата и время запроса
- Два варианта подтверждения:
  - **Без оплаты**: немедленное подтверждение и активация подписки
  - **С оплатой**: отправка реквизитов клиенту и ожидание чека

#### 3.1.4 Проверка платежей
- Очередь полученных чеков от клиентов
- Просмотр чека (изображение или PDF)
- Информация о платеже:
  - Клиент
  - Сумма
  - Услуга
  - Дата и время отправки
- Действия: подтвердить оплату/отклонить
- После подтверждения - автоматическая активация подписки

#### 3.1.5 Управление шкафами
- Общее количество шкафов
- Статус каждого шкафа (свободен/занят)
- Номер шкафа
- Клиент, занимающий шкаф
- Срок аренды
- Дата окончания аренды

#### 3.1.6 Управление подписками
- Список всех активных подписок
- Фильтрация по типу (игра/шкаф)
- Фильтрация по клиенту
- Информация о подписке:
  - Тип услуги
  - Дата начала
  - Дата окончания (для срочных подписок)
  - Стоимость
  - Статус оплаты
- Возможность продления/отмены подписки

#### 3.1.7 Настройки системы
- Реквизиты для оплаты (номера карт)
- Контактный номер телефона для связи (отправляется новым клиентам)
- Тарифы на услуги
- Общее количество шкафов

### 3.2 Telegram-бот для клиентов

#### 3.2.1 Авторизация
- Запрос номера телефона при первом запуске
- Формат ввода: +998 xx xxx-xx-xx
- Проверка номера в базе данных:
  - **Если номер существует**: автоматический вход без подтверждения
  - **Если номер новый**: режим ожидания подтверждения администратором
- После отправки запроса на регистрацию клиенту приходит автоматическое сообщение с контактным номером для связи

#### 3.2.2 Главное меню (для подтвержденных клиентов)
Основные разделы:
- Мои подписки
- Забронировать услугу
- Профиль
- Связаться с администрацией

#### 3.2.3 Раздел "Мои подписки"
Отображение всех активных подписок клиента:
- **Подписка на игру**:
  - Тип (единоразовая/ежемесячная)
  - Дата начала
  - Дата окончания (если применимо)
  - Статус
- **Бронирование шкафа**:
  - Номер шкафа
  - Дата начала аренды
  - Дата окончания аренды
  - Стоимость в месяц

#### 3.2.4 Процесс бронирования
**Шаг 1: Выбор услуги**
- Подписка на игру
- Бронирование шкафа
- Комплексный пакет (игра + шкаф)

**Шаг 2: Выбор опций**
- Для игры: единоразовая или ежемесячная подписка
- Для шкафа: выбор срока (минимум 1 месяц)
- Расчет стоимости

**Шаг 3: Проверка доступности**
- Если выбрано бронирование шкафа: автоматическая проверка наличия свободных шкафов
- Если шкафы заняты: сообщение о недоступности
- Если шкафы доступны: возможность отправить запрос

**Шаг 4: Отправка запроса**
- Подтверждение выбранных услуг и стоимости
- Отправка запроса администратору
- Сообщение о том, что запрос находится в обработке

**Шаг 5: Ожидание решения администратора**
- Два сценария:
  - **Подтверждение без оплаты**: немедленная активация подписки, уведомление клиента
  - **Запрос оплаты**: получение реквизитов карты

**Шаг 6: Оплата (если требуется)**
- Получение номера карты для перевода
- Инструкция по оплате
- Запрос на отправку чека
- Возможность загрузить файл (PDF, JPG, PNG)
- Сообщение о том, что чек отправлен на проверку

**Шаг 7: Подтверждение оплаты**
- Ожидание проверки администратором
- Уведомление о подтверждении/отклонении
- При подтверждении: активация подписки

#### 3.2.5 Уведомления
Клиент получает уведомления о:
- Статусе регистрации (принят/отклонен)
- Статусе запроса на бронирование
- Получении реквизитов для оплаты
- Статусе проверки чека
- Активации подписки
- Истечении срока подписки (за 3 дня)
- Продлении подписки

## 5. Технические требования

### 5.1 Технологический стек

#### Backend
- **Framework**: Laravel (текущая версия проекта)
- **Admin Panel**: Laravel Orchid
- **Telegram Bot SDK**: [irazasyed/telegram-bot-sdk](https://github.com/irazasyed/telegram-bot-sdk)
- **Database**: MySQL/PostgreSQL
- **PHP Version**: 8.1+

#### Дополнительные технологии
- Queue system для обработки уведомлений (Laravel Queue)
- Storage для хранения чеков (Laravel Storage)
- Validation для проверки данных
- Events и Listeners для обработки событий

### 5.2 Структура базы данных

#### Основные таблицы

**users** (администраторы)
- id
- name
- email
- password
- created_at, updated_at

**clients** (клиенты гольф-клуба)
- id
- phone_number (уникальный, формат: +998 xx xxx-xx-xx)
- telegram_id (уникальный)
- first_name
- last_name
- username (Telegram)
- status (pending/approved/blocked)
- approved_by (связь с users)
- approved_at
- created_at, updated_at

**lockers** (шкафы)
- id
- locker_number
- status (available/occupied)
- created_at, updated_at

**subscriptions** (подписки)
- id
- client_id (связь с clients)
- subscription_type (game_once/game_monthly/locker)
- locker_id (связь с lockers, nullable)
- start_date
- end_date (nullable для ежемесячных)
- price
- status (active/expired/cancelled)
- created_at, updated_at

**booking_requests** (запросы на бронирование)
- id
- client_id (связь с clients)
- service_type (game/locker/both)
- game_subscription_type (once/monthly, nullable)
- locker_duration_months (nullable)
- total_price
- status (pending/payment_required/payment_sent/approved/rejected)
- processed_by (связь с users, nullable)
- processed_at (nullable)
- created_at, updated_at

**payments** (платежи и чеки)
- id
- booking_request_id (связь с booking_requests)
- client_id (связь с clients)
- amount
- receipt_file_path
- status (pending/verified/rejected)
- verified_by (связь с users, nullable)
- verified_at (nullable)
- created_at, updated_at

**settings** (настройки системы)
- id
- key
- value
- description
- created_at, updated_at

#### Связи между таблицами
- clients → subscriptions (один ко многим)
- clients → booking_requests (один ко многим)
- clients → payments (один ко многим)
- lockers → subscriptions (один ко многим)
- booking_requests → payments (один к одному)
- users → clients (один ко многим, для approved_by)

## 6. Бизнес-логика и правила

### 6.1 Правила регистрации
- Один номер телефона = один аккount
- Новые клиенты требуют подтверждения администратора
- Существующие клиенты входят автоматически
- При регистрации отправляется автоматическое сообщение с контактными данными

### 6.2 Правила бронирования
- Минимальный срок аренды шкафа: 1 месяц
- Фиксированная цена аренды шкафа: $10/месяц
- Проверка доступности шкафов перед бронированием
- Клиент может иметь одновременно подписку на игру и аренду шкафа
- Запрос может быть подтвержден с оплатой или без

### 6.3 Правила оплаты
- Администратор решает, требуется ли оплата
- Если оплата требуется:
  - Клиенту отправляются реквизиты карты
  - Клиент отправляет чек (PDF, JPG, PNG)
  - Администратор проверяет и подтверждает
- Если оплата не требуется:
  - Подписка активируется немедленно

### 6.4 Правила управления подписками
- Подписки имеют статусы: active, expired, cancelled
- Система должна отслеживать истечение сроков
- Уведомления о скором истечении подписки (за 3 дня)
- При истечении месячной подписки на игру требуется продление
- При истечении аренды шкафа - шкаф освобождается

### 6.5 Правила управления шкафами
- Каждый шкаф имеет уникальный номер
- Шкаф может быть либо свободен, либо занят
- При активации подписки на шкаф - автоматическое назначение свободного шкафа
- При истечении аренды - автоматическое освобождение шкафа
- В админке должна быть видна информация о занятости шкафов

## 7. Пользовательские сценарии (Use Cases)

### 7.1 Регистрация нового клиента

**Актор**: Новый клиент

**Предусловия**: Клиент запускает Telegram-бота

**Основной сценарий**:
1. Клиент отправляет команду /start
2. Бот запрашивает номер телефона
3. Клиент отправляет номер телефона
4. Система проверяет номер в базе данных
5. Номер не найден - создается запрос на регистрацию
6. Клиент получает сообщение об ожидании подтверждения
7. Клиент получает автоматическое сообщение с контактным номером для связи
8. Администратор получает уведомление о новом запросе
9. Администратор подтверждает регистрацию
10. Клиент получает уведомление об успешной регистрации
11. Клиент получает доступ к главному меню

**Альтернативный сценарий**:
- Шаг 4а: Номер найден в базе - переход к шагу 11
- Шаг 9а: Администратор отклоняет регистрацию - клиент получает уведомление об отказе

### 7.2 Бронирование подписки на игру

**Актор**: Подтвержденный клиент

**Предусловия**: Клиент авторизован и находится в главном меню

**Основной сценарий**:
1. Клиент выбирает "Забронировать услугу"
2. Бот предлагает выбрать тип услуги
3. Клиент выбирает "Подписка на игру"
4. Бот предлагает выбрать тип подписки (единоразовая/ежемесячная)
5. Клиент выбирает тип
6. Бот показывает стоимость и запрашивает подтверждение
7. Клиент подтверждает
8. Запрос отправляется администратору
9. Клиент получает сообщение об обработке запроса
10. Администратор выбирает подтверждение без оплаты
11. Подписка активируется автоматически
12. Клиент получает уведомление об активации

**Альтернативный сценарий 1** (с оплатой):
- Шаг 10а: Администратор выбирает подтверждение с оплатой
- Клиент получает реквизиты карты
- Клиент переводит деньги и отправляет чек
- Администратор проверяет чек
- Администратор подтверждает оплату
- Подписка активируется
- Клиент получает уведомление

**Альтернативный сценарий 2** (отклонение):
- Шаг 10б: Администратор отклоняет запрос
- Клиент получает уведомление об отказе

### 7.3 Бронирование шкафа

**Актор**: Подтвержденный клиент

**Предусловия**: Клиент авторизован, есть свободные шкафы

**Основной сценарий**:
1. Клиент выбирает "Забронировать услугу"
2. Бот предлагает выбрать тип услуги
3. Клиент выбирает "Бронирование шкафа"
4. Бот предлагает выбрать срок (минимум 1 месяц)
5. Клиент выбирает количество месяцев
6. Система проверяет наличие свободных шкафов
7. Свободные шкафы есть
8. Бот показывает стоимость и запрашивает подтверждение
9. Клиент подтверждает
10. Запрос отправляется администратору
11. Администратор обрабатывает запрос (с оплатой или без)
12. При подтверждении - автоматическое назначение свободного шкафа
13. Клиент получает уведомление с номером шкафа

**Альтернативный сценарий**:
- Шаг 6а: Свободных шкафов нет
- Клиент получает сообщение о недоступности
- Процесс завершается

### 7.4 Комплексное бронирование (игра + шкаф)

**Актор**: Подтвержденный клиент

**Предусловия**: Клиент авторизован, есть свободные шкафы

**Основной сценарий**:
1. Клиент выбирает "Забронировать услугу"
2. Бот предлагает выбрать тип услуги
3. Клиент выбирает "Комплексный пакет (игра + шкаф)"
4. Бот предлагает выбрать тип подписки на игру
5. Клиент выбирает тип
6. Бот предлагает выбрать срок аренды шкафа
7. Клиент выбирает количество месяцев
8. Система проверяет наличие свободных шкафов
9. Свободные шкафы есть
10. Бот показывает общую стоимость и запрашивает подтверждение
11. Клиент подтверждает
12. Запрос отправляется администратору
13. Администратор обрабатывает запрос
14. При подтверждении - активация обеих подписок
15. Клиент получает уведомление с деталями

### 7.5 Просмотр активных подписок

**Актор**: Подтвержденный клиент

**Предусловия**: Клиент авторизован

**Основной сценарий**:
1. Клиент выбирает "Мои подписки"
2. Бот запрашивает активные подписки из базы данных
3. Бот отображает список подписок с деталями:
   - Тип подписки
   - Дата начала
   - Дата окончания (если применимо)
   - Номер шкафа (если есть)
   - Статус
4. Клиент просматривает информацию

**Альтернативный сценарий**:
- Шаг 2а: У клиента нет активных подписок
- Бот отображает сообщение об отсутствии подписок
- Бот предлагает забронировать услугу
