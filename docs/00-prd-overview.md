## 1. Введение

### 1.1 Цель проекта
Создание автоматизированной системы для управления подписками гостей гольф-клуба, включающей:
- **Администраторскую панель** (Laravel Orchid) для управления клиентами, подписками и платежами
- **Telegram-бот** для клиентов - бронирование услуг и управление подписками

### 1.2 Текущее состояние проекта
- ✅ Фреймворк Laravel установлен
- ✅ Пакет Laravel Orchid настроен
- ✅ Пакет `irazasyed/telegram-bot-sdk` установлен
- ❌ Бизнес-логика не реализована
- ❌ База данных не спроектирована
- ❌ Telegram-бот не создан

### 1.3 Ожидаемые результаты
После завершения проекта будет функционировать:
1. Полноценная админ-панель для управления всеми аспектами гольф-клуба
2. Telegram-бот для клиентов с возможностью регистрации, бронирования и оплаты
3. Система автоматических уведомлений
4. Система управления шкафами для оборудования

## 2. Бизнес-контекст

### 2.1 Услуги гольф-клуба

| Услуга | Тип | Стоимость | Описание |
|--------|-----|-----------|----------|
| Подписка на игру | Единоразовая | Настраиваемая | Одноразовый доступ к полю |
| Подписка на игру | Ежемесячная | Настраиваемая | Регулярный месячный доступ |
| Аренда шкафа | Ежемесячная | $10/месяц | Минимум 1 месяц |

### 2.2 Типы пользователей

#### Администраторы
**Роль**: Управление системой и обработка запросов клиентов

**Возможности**:
- Просмотр и управление базой клиентов
- Подтверждение/отклонение регистраций новых клиентов
- Обработка запросов на бронирование (с оплатой или без)
- Проверка платежных чеков
- Управление шкафами и их статусами
- Мониторинг всех активных подписок
- Настройка системных параметров (тарифы, реквизиты)

#### Клиенты (гости гольф-клуба)
**Роль**: Использование услуг клуба через Telegram-бот

**Возможности**:
- Регистрация через Telegram-бот
- Просмотр своих активных подписок
- Бронирование услуг (игра, шкаф, комплекс)
- Отправка чеков об оплате
- Получение уведомлений о статусах

## 4. Пользовательские сценарии (Use Cases)

### UC-01: Регистрация нового клиента

| Поле | Значение |
|------|----------|
| **Актор** | Новый клиент |
| **Предусловие** | Клиент установил Telegram |
| **Триггер** | Отправка команды /start боту |

**Основной поток**:
1. Клиент отправляет `/start`
2. Бот запрашивает номер телефона (кнопка "Поделиться контактом")
3. Клиент отправляет номер
4. Система проверяет номер в БД
5. Номер не найден → создается запрос на регистрацию
6. Клиент видит: "Ваш запрос отправлен на рассмотрение"
7. Клиент получает контактный номер администрации
8. Администратор получает уведомление
9. Администратор подтверждает регистрацию
10. Клиент получает уведомление об успехе
11. Клиент получает доступ к главному меню

**Альтернативный поток A**: Номер уже в базе
- На шаге 4: номер найден → автоматический вход (шаг 11)

**Альтернативный поток B**: Отклонение
- На шаге 9: администратор отклоняет → клиент получает отказ

---

### UC-02: Бронирование подписки на игру

| Поле | Значение |
|------|----------|
| **Актор** | Подтвержденный клиент |
| **Предусловие** | Клиент авторизован в боте |
| **Триггер** | Выбор "Забронировать услугу" |

**Основной поток**:
1. Клиент выбирает "Забронировать услугу"
2. Бот показывает типы услуг
3. Клиент выбирает "Подписка на игру"
4. Бот показывает типы подписок (единоразовая/ежемесячная)
5. Клиент выбирает тип
6. Бот показывает стоимость и кнопку подтверждения
7. Клиент подтверждает
8. Запрос отправляется администратору
9. Клиент видит: "Запрос в обработке"
10. Администратор подтверждает без оплаты
11. Подписка активируется
12. Клиент получает уведомление об активации

---

### UC-03: Бронирование шкафа

| Поле | Значение |
|------|----------|
| **Актор** | Подтвержденный клиент |
| **Предусловие** | Есть свободные шкафы |
| **Триггер** | Выбор "Забронировать услугу" |

**Основной поток**:
1. Клиент выбирает "Забронировать услугу"
2. Клиент выбирает "Бронирование шкафа"
3. Бот предлагает выбрать срок (1, 3, 6, 12 месяцев)
4. Клиент выбирает количество месяцев
5. Система проверяет наличие свободных шкафов
6. Свободные шкафы есть
7. Бот показывает стоимость ($10 × месяцы)
8. Клиент подтверждает
9. Запрос отправляется администратору
10. При подтверждении → автоматическое назначение шкафа
11. Клиент получает номер шкафа

**Альтернативный поток**: Нет свободных шкафов
- На шаге 5-6: нет шкафов → сообщение о недоступности

---

### UC-04: Оплата бронирования

| Поле | Значение |
|------|----------|
| **Актор** | Клиент |
| **Предусловие** | Администратор запросил оплату |
| **Триггер** | Получение реквизитов |

**Основной поток**:
1. Клиент получает номер карты для перевода
2. Клиент переводит деньги
3. Клиент отправляет боту чек (фото/PDF)
4. Бот подтверждает получение чека
5. Администратор проверяет чек
6. Администратор подтверждает оплату
7. Подписка активируется
8. Клиент получает уведомление

## 5. Системные настройки

### 5.1 Настраиваемые параметры

| Параметр | Тип | Описание |
|----------|-----|----------|
| `payment_card_number` | string | Номер карты для оплаты |
| `contact_phone` | string | Контактный телефон администрации |
| `game_once_price` | decimal | Цена единоразовой игры |
| `game_monthly_price` | decimal | Цена ежемесячной подписки |
| `locker_monthly_price` | decimal | Цена аренды шкафа ($10) |
| `total_lockers_count` | integer | Общее количество шкафов |
| `notification_days_before` | integer | За сколько дней уведомлять (3) |

### 5.2 Формат данных

| Данные | Формат | Пример |
|--------|--------|--------|
| Телефон | `+998 XX XXX-XX-XX` | `+998 90 123-45-67` |
| Telegram ID | integer | `123456789` |
| Дата | `YYYY-MM-DD` | `2024-01-15` |
| Сумма | decimal(10,2) | `100.00` |

## 6. Нефункциональные требования

### 6.1 Производительность
- Время отклика API: < 2 секунд
- Время ответа бота: < 3 секунд
- Поддержка: до 1000 активных клиентов

### 6.2 Безопасность
- Авторизация администраторов по логину/паролю
- Валидация всех входных данных
- Защита от SQL-инъекций (Eloquent ORM)
- Защита от CSRF (Laravel middleware)
- Шифрование паролей (bcrypt)

### 6.3 Надежность
- Логирование критических операций
- Обработка ошибок без падения системы
- Резервное копирование БД

### 6.4 Масштабируемость
- Модульная архитектура
- Использование очередей для уведомлений
- Возможность горизонтального масштабирования
